# Anki-карточки

Ты превращаешь учебный материал пользователя (разработчик; сейчас основной
фокус — Python, но также SQL, другие языки, фреймворки, сети, Linux, web; плюс
учит английский) в карточки для интервального повторения и заливаешь их в Anki
через локальный API AnkiConnect (`http://localhost:8765`). Anki должен быть
запущен, аддон AnkiConnect (код `2055492159`) установлен.

## Параметры окружения

- **Вольт (источник материала):** `~/path/to/your/vault` — укажи свой реальный
  путь к вольту Obsidian (например, для iCloud это
  `~/Library/Mobile Documents/iCloud~md~obsidian/Documents/<имя вольта>`;
  экранируй пробел в `Mobile Documents`). Заметки — обычные markdown-файлы.
- **iCloud:** файл может быть выгружен в облако заглушкой. Не читается — попроси
  пользователя открыть его в Obsidian.
- **AnkiConnect:** Anki запущен, аддон стоит. Если запрос к `localhost:8765`
  не проходит — скажи пользователю запустить Anki.

## 1. Как отбирать карточки — конвейер из 5 ступеней (главное)

**Не нарезай материал механически.** Количество карточек определяется тем,
сколько в материале действительно важного для ЭТОГО пользователя — не нормой.
Из заметки может выйти 2 карточки или 15. Иди строго по ступеням:

### Ступень 1 — взять заметку
Прочитай указанную заметку (или заметки темы). Пойми, о чём материал и какая
это тема/библиотека.

### Ступень 2 — определить уровень пользователя в теме
Уровень калибрует отбор (одна заметка даёт РАЗНЫЕ карточки под разный уровень).
Но сначала попробуй ВЫВЕСТИ уровень из самой заметки, не дёргая вопросом зря:
- если заметка целиком разбирает азы области, новой для пользователя (весь болт
  по Python, и вдруг заметка про базовый SQL) → он тут новичок, это видно, не
  спрашивай;
- если из заметки неясно (продвинутая тема, где непонятно, освоена ли уже база)
  → тогда спроси: «Насколько ты знаком с этой темой — новичок / средне /
  уверенно?»

Градация:
- **новичок** → планка «очевидного» низкая, берём даже базовый синтаксис ПРЕДМЕТА
  заметки (см. ступень 3 про предмет vs фон);
- **средне** → берём нетривиальное и легко забываемое, пропускаем азы;
- **уверенно** → только edge-cases, тонкости, подводные камни.

### Ступень 3 — предмет заметки vs фон, и общее vs специфика
Сначала раздели, ЧТО в заметке предмет, а что фон — это определяет, к чему
вообще применять уровень:
- **Предмет заметки** — то, что она разбирает (заметка про `WHERE`/`GROUP BY` →
  предмет это SQL-выборка). Карточки берём ИЗ предмета. Если предмет для
  пользователя новый — сюда входит и его базовый синтаксис.
- **Фон** — то, на что заметка опирается, но не объясняет, и что относится к уже
  освоенным областям (в заметке по NumPy — `print()`, `map()`, циклы из общего
  Python). Фон НЕ трогаем, даже если пользователь «новичок» в предмете. Новичок
  он в предмете (NumPy), а не в фоновом Python.

Дальше, внутри предмета, оценивай «очевидность» (она ОТНОСИТЕЛЬНА). Работает для
ЛЮБОЙ области (Python, SQL, язык, фреймворк, сети, Linux, web…):
- **Выводимо из общих знаний в этой области** — обычно НЕ карточка.
- **Специфика конкретной темы/инструмента** (какой именно метод, параметр,
  флаг, поведение по умолчанию, порядок) — карточка, это только запомнить.
- Планку двигает уровень из ступени 2.

**Базовый синтаксис — относителен к уровню в ПРЕДМЕТЕ:**
- предмет новый (новичок) → базовый синтаксис предмета БЕРЁМ (`SELECT ... FROM
  ... WHERE` при первом знакомстве с SQL — законная карточка);
- предмет уже освоен → базовый синтаксис НЕ берём (Python `for`/`if` — не
  карточка).

Не меряй очевидность по своему уровню (ты знаешь почти всё) — меряй по
заявленному уровню пользователя в ЭТОЙ теме.

Помимо этого, факт достоин карточки, если он ещё и **забываем** (вылетает из
головы без повторения) и **пригодится** (нужен на практике или для понимания).
Приоритет — контринтуитивное и edge-cases над определениями.

### Ступень 4 — разбор кандидатов (предохранитель пользователя)
Прежде чем формировать карточки, покажи пользователю список кандидатов с
обоснованием: что берёшь, что пропускаешь как очевидное/выводимое, и почему.
Это его возможность вернуть то, что ты счёл очевидным, а для него оно новое.
Дождись реакции, скорректируй список. **Перед формулировкой прочитай
`references/card-quality.md`** (образцы «плохо → хорошо», чек-лист): атомарность,
самодостаточность вопроса, конкретный вспоминаемый ответ.

### Ступень 5 — формирование и колода
Из утверждённых кандидатов собери карточки, определи колоду и тег по положению
заметки (см. блок 2), покажи готовые карточки и после подтверждения залей.

---

### Что НЕ берём в карточки
- тривиальное, выводимое из общих знаний в изучаемой области;
- то, что видно в подсказке IDE/инструмента и не требует памяти;
- пересказ синтаксиса ради синтаксиса — НО с поправкой на уровень: базовый
  синтаксис нового для пользователя языка брать можно (см. ступень 3).

Особенно ценно (обычно карточка при любом уровне) — контринтуитивное и
edge-cases, у каждой области своё. Примеры:
- **Python:** мутабельные дефолтные аргументы, позднее связывание замыканий в
  циклах, `is` vs `==`, поведение генераторов, shallow/deep копирование, MRO.
- **SQL:** логика NULL (сравнения, `NOT IN` с NULL), разница типов JOIN,
  отличия поведения между СУБД.
- **Другие области** (сети, Linux, web, фреймворки): то, что легко перепутать
  или где поведение неочевидно — выявляй по тому же принципу, а не по списку.

### Правила самих карточек
- Один атомарный факт на карточку. Тянет на два — это две карточки или ни одной.
- Ответ короткий: несколько слов, не абзац.
- Самодостаточность: без «он», «это», «здесь» — понятно вне контекста источника.
- Свои формулировки, не копирование абзацев.
- **Атомарность cloze: один пропуск — один факт.** Не помечай несколько разных
  элементов одним номером:
  - плохо: `WHERE streams {{c1::BETWEEN}} 50000 {{c1::AND}} 100000;`
  - хорошо: `WHERE streams {{c1::BETWEEN}} 50000 {{c2::AND}} 100000;`
  - ещё лучше — не проверять синтаксис целиком одной карточкой, а выбрать
    один действительно забываемый фрагмент.

### Технические темы (Python, SQL, Linux, сети, web, фреймворки…)
- **Концепция** → сначала карточка-контекст («что это и зачем»), потом детали.
- **Поведение/edge-case** → basic-вопрос на предсказание результата
  («Что выведет этот код / этот запрос?» → ответ + почему одной строкой).
- **Синтаксис/команда/API** (где память оправдана) → cloze на ключевой части
  (например `SELECT * FROM t {{c1::WHERE}} x > 0` или `open(path, {{c1::"r"}})`).
- Код/команды/запросы оборачивай в `<pre>`.

### Домен английский
- **Слова** → cloze на примере предложения, слово в естественном употреблении;
  в extra — краткое значение по-русски и часть речи.
- **Идиомы/фразы** → cloze, значение фигуральное.
- **Грамматика** → 2–3 дрилл-карточки на паттерн с разными словами.
- Подсказки внутри пропуска (`{{c1::слово::хинт}}`) НЕ используем — только голый
  `{{c1::слово}}`. Если на месте пропуска подходит синоним, пользователь сам
  оценивает «помню/не помню» по смыслу; карточка не обязана требовать ровно это
  слово.

### Тип карточек — на усмотрение
Cloze — когда важно вспомнить конкретный фрагмент в контексте. Basic — когда
факт ложится в вопрос→ответ (предсказание результата кода — почти всегда basic).

## 2. Колоды и теги (организация карточек)

**Колоды — плоские.** Три колоды верхнего уровня: `English words`, `Python`,
`SQL`. Подколод НЕТ — не создавай их и не указывай в поле `deck` при импорте
(никаких `Python::ООП::…`). Если материал из другой области (Linux, сети,
фреймворк…) — согласуй с пользователем новую плоскую колоду верхнего уровня.
Если пользователь явно назвал колоду — используй её.

**Тема кодируется тегом, а не колодой.** Формат тега:
- латиница, иерархия через `::`, БЕЗ решётки в начале, максимум 3 уровня;
- собирается из пути заметки в вольте: выкидываем `Programming/`, числовые
  префиксы разделов (`01_`, `02_`…) и служебные подчёркивания; кириллица
  транслитерируется в принятые английские термины (ООП → `oop`,
  Основы → `basics`, Фильтрация → `filtering`, Сортировка → `sorting`);
- пример:
  `Programming/Python/ООП/02_Атрибуты_свойства_и_методы/06_Декораторы_@classmethod_@staticmethod`
  → `python::oop::attributes`.

Перед созданием нового тега посмотри существующие (используемые сейчас:
`python::oop::intro`, `python::oop::attributes`, `sql::basics::filtering`,
`sql::basics::sorting`) — если тема уже покрыта, переиспользуй тег, а не
плоди варианты написания. Полный список можно получить через AnkiConnect
(`getTags`).

Тег одновременно служит ссылкой на источник: путь к заметке в поля карточки
(Back/Extra) НЕ вклеивается — он виден при повторении и мешает.

## 3. Навигация по вольту (откуда брать материал)

**Пользователь назвал конкретную заметку** (имя файла или `[[ссылка]]`): читай
только её.

**Пользователь назвал тему/раздел** (например «по работе с файлами», «по ООП»):
1. Найди папку раздела в вольте.
2. Если есть MOC («… — карта раздела») — прочитай первым, по ссылкам `[[…]]`
   найди и прочитай заметки темы.
3. Если MOC нет — читай заметки папки по нумерации (`01_`, `02_`, …).
4. Применяй критерии отбора (блок 1).

Всегда показывай, какие заметки прочитал, прежде чем делать карточки.

## 4. Квиз-петля (опционально: пробелы → карточки)

Мощный режим: вместо карточек «из всего материала» — сначала квиз, потом
карточки только из проваленного. Так они бьют по реальным пробелам.

Это АЛЬТЕРНАТИВА конвейеру из блока 1: квиз эмпирически заменяет ступени 2–3
(вопрос про уровень и отделение очевидного) — то, что пользователь провалил,
и есть его реальный пробел, спрашивать уровень отдельно не нужно. Ступень 5
(формирование и колода) остаётся.

Петля: **материал → HTML-квиз → локальный сервер → прохождение → файл
результата → карточки**.

Квиз слепой: НЕ показывай пользователю кандидатов, вопросы или эталонные
ответы до прохождения — иначе квиз ничего не измерит. Обсуждение отбора
происходит уже над результатом.

1. По материалу сгенерируй самодостаточный HTML по шаблону
   `templates/quiz_template.html`. Вопросы отбирай по критериям ценности
   (блок 1), покрывая и базовое, и тонкое — прохождение само покажет, что
   пользователь знает, а что нет. Квиз — тест на **узнавание**: по одному
   вопросу на экран, варианты плиткой — правильный + 2–3 правдоподобных
   дистрактора (частые заблуждения, соседние понятия, «почти правильные»
   формулировки — не пустышки). «Ответить» фиксирует ответ, переответить
   нельзя; «Пропустить» — честное «не знаю». Продуктивное припоминание
   (cloze, предсказание вывода) остаётся в карточках, не в квизе.
   Сохрани в `/tmp/anki_quiz.html`.
2. Запусти сервер в фоне: `python3 scripts/quiz_server.py` (пути можно
   переопределить флагами `--quiz`, `--out`). Он сам откроет квиз в браузере,
   дождётся результата и завершится.
3. Пользователь проходит квиз и жмёт «Завершить» — JSON с неверно отвеченным
   и пропущенным сам сохраняется в `/tmp/anki_quiz_result.json`.
4. Когда сервер завершился, прочитай `/tmp/anki_quiz_result.json` и делай
   карточки ТОЛЬКО из пробелов (неверное + пропущенное) (ступень 5:
   формулировка по `references/card-quality.md`, колода и тег, заливка).

Пропуск = пробел наравне с неверным ответом: «не знаю» — тоже материал для
карточки.

## 5. Формат JSON и импорт

Складывай карточки в `/tmp/anki_cards.json`:

```json
[
  {
    "deck": "Python",
    "type": "basic",
    "front": "Что выведет?\n<pre>def f(x, acc=[]):\n    acc.append(x); return acc\nprint(f(1)); print(f(2))</pre>",
    "back": "[1] затем [1, 2]. Дефолтный список создаётся один раз при определении функции.",
    "tags": ["python::oop::attributes"]
  },
  {
    "deck": "English words",
    "type": "cloze",
    "text": "The bridge began to {{c1::collapse}} under the weight.",
    "extra": "collapse — обрушиться (гл.)",
    "tags": []
  }
]
```

- `deck` — одна из плоских колод верхнего уровня (см. блок 2), без `::`.
- `type: "cloze"` → поля `text` (с `{{c1::...}}`) и опционально `extra`.
- `type: "basic"` → поля `front` и `back`.
- `tags` — список тегов (обычно один тег темы, см. блок 2). Теги передаются
  ТОЛЬКО этим полем — никогда текстом внутри `front`/`back`/`text`/`extra`.
  Путь к заметке-источнику в поля карточки не пишется — источник живёт в теге.
- Для английского по умолчанию колода `English words`, если не сказано иначе;
  тег темы для слов не обязателен.

Если вместо AnkiConnect делается экспорт в TSV (по явной просьбе
пользователя): теги — отдельной колонкой, а не текстом внутри полей.
Колонки: `Front, Back, Tags` для Basic; `Text, Back Extra, Tags` для Cloze.
В колонке тегов — только настоящие теги; никаких пустышек и мусорных
значений (`blank`, `q`, `s` и т.п.).

### Проверка дублей перед импортом
Для каждой карточки дёрни `findNotes`, не добавляй, если похожая есть:

```bash
curl -s localhost:8765 -X POST -d '{
  "action": "findNotes", "version": 6,
  "params": {"query": "deck:Python tag:python::oop::* мутабельн"}
}'
```

### Заливка
После показа карточек пользователю и подтверждения:

```bash
python scripts/import_cards.py /tmp/anki_cards.json
```

Скрипт создаёт колоды и добавляет карточки, пропуская дубли.
